Policy Config UI v6

Three approaches to policy config with DeSlop as the default policy. All built against the 5 surviving requirements from Round 2.

Design brief: Users decide in one session whether Luthien is worth their time (R1). The default policy must show obvious value on the first request. DeSlop (string replacement - removes em-dashes, curly quotes, AI-isms) is the default because it produces a visible change on every response.

Requirements addressed: R1 (obvious first value), R2 (interact from terminal or browser), R3 (verify policies working), R4 (customize for context), R5 (see what's active, toggle on/off with progressive disclosure).
A Toggles + Blocked Log

Minimal dashboard. DeSlop is on by default. Policies listed as toggle switches. Below: a log of what each policy changed or blocked. Two-step Claude Code setup at the top.

R1 default on R2 setup instructions R3 blocked log R5 toggles
  • Simplest to build and maintain
  • Jack validated this pattern (v3B)
  • Progressive disclosure: toggle first, config later
  • No live preview of what DeSlop does
  • Customization (R4) requires expanding a panel
  • Blocked log may be empty on first visit

B Live Demo First

C Status Board

Read-only status view. Shows what policies are active, what they changed in the last N requests, and aggregate stats. All configuration happens in YAML or CLI. The web page is for verification (R3), not configuration.

R1 default active, stats visible R2 config stays in terminal R3 verification dashboard R5 shows what's active
  • Matches terminal-first users (Jack, Zac)
  • No risk of building config UI nobody uses
  • Simplest web surface, easiest to maintain
  • No in-browser customization (R4 gap)
  • Harder to demo to non-technical stakeholders
  • Two places to look: terminal for config, browser for status
