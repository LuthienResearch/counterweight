## Session 1: Zac Saber (Counterweight AI)

- **Date:** ~Jan 30, 2026 (Thursday)
- **Forum:** In-person at Mox, SF
- **Attendees:** Scott Wofford, Jai Dhyani, Zac Saber
- **CX version:** Policy Configuration page with SimpleJudgePolicy active, full list of available policies (EventBasedNoOpPolicy, SimpleJudgePolicy, DeSlop, AllCapsPolicy, BlockToolCallsPolicy), test message panel, and config fields (judge_prompt, threshold, judge_model)

### Feedback

1. **"It's kind of overwhelming and hard to use."** Too many options on the page without clear guidance on what to look at or do first.

2. **Wants meaningful default policy, not AllCaps.** "AllCaps as a user is not meaningful to me. 'I'm going to block jailbreaks' is meaningful." Default should demonstrate real value, not just prove the system works.

3. **Doesn't care about the test panel as a user.** "This is good for you guys as a debugging thing. For me, I would be using the product [in Claude Code]." The test message UI is useful for Luthien's internal debugging but not the user's primary workflow.

4. **Wants a direct path from config → Claude Code.** "It should say right here, here's how you can test with Claude Code." After configuring a policy, the next step should be obvious: go use it in Claude Code and watch the policy activate.

5. **Don't tell me what to type.** "If you tell me what to type, I'll be like, this is already not my work." Instructions should point to Claude Code, not script the interaction.

6. **Simplify the visible config.** "API base and API key... can be hidden for me. Obviously I need to adjust [the threshold] because it's the only parameter." Hide what's not user-relevant. Show only what the user needs to decide.

7. **Threshold needs better documentation.** When threshold was set to 0.7, nothing was blocked. Jai had to manually suggest changing to 0.1. The threshold parameter should be more prominent with clear explanation of what values mean.

8. **Preferred notification style: silent blocking with async report.** "I want to run Claude Code by itself and just let it run. I want it to be blocked on stuff which is scope creepy. When it's finished... that's when I want to see what got rejected. I don't care about it till then."

9. **Doesn't want a custom website/URL for reports.** "I'd probably just want an email or some Slack channel... if I have to visit a custom website for it, then I might not visit that."

10. **Auto-retry on blocked actions.** Wants Claude to automatically retry when something is blocked, rather than stopping and waiting for user input.

### Zac's additional feedback (Signal, Jan 31)

11. **Wants policy-as-automated-QA.** "I'd prefer that at the end of this loop, a policy runs which evaluates the original prompt ('match streamlit UX') against the final agent message ('I deviated from UX because I think I know better') and send a followup: 'check that your final reasoning matches user's original request.'"

## Session 4: Bahar Erar (enterprise AI safety perspective)

- **Date:** Feb 8, 2026 (Super Bowl Sunday)
- **Forum:** In-person, Super Bowl party
- **Attendees:** Scott Wofford, Bahar Erar
- **Bahar:** Former Amazon Applied Scientist — managed AI safety/policy compliance at enterprise scale (millions of conversations). Not a daily Claude Code user.
- **CX version:** README options C and A + live terminal demo of PipBlockPolicy. Bahar also watched the Luthien blocking demo video.
- **Full session:** [1-value-prop-feedback.md, Session 2](1-value-prop-feedback.md#session-2-bahar-erar) — this summary covers only the policy-relevant findings.

### Feedback on policy creation

1. **"Write your own" is too rudimentary.** "The problem usually of setting these policies is not to define it — but to test it against all the different edge cases, and then actually refining the policy is the hardest part." A 5-line Python class undersells the real work.

2. **"I'm not gonna allow drunk people."** "Everybody can say 'I'm not gonna allow drunk people.' How do you define that?" The policy statement is easy; operationalizing it is hard.

3. **Policy = knowledge + detection + enforcement.** Three parts: (a) system must be aware of policies/rules that change, (b) detect when a response relates to a policy, (c) ensure the response refers to/complies with the policy. "You do need to modularize the solution a little bit."

### Feedback on universal vs custom policies

4. **Handle fundamentals so I can focus on domain-specific.** "When I was working on my team, what I want to focus on is the things that are unique to our space — because those are the hardest things." Luthien should provide universal safety defaults.

5. **LLM-as-judge is already common.** "A lot of enterprise scale applications are doing LLM as a judge themselves." Challenge: how is Luthien different from what teams already build internally? (Answer: Luthien provides the infrastructure + universal defaults; you bring the domain rules.)

6. **Seller review manipulation example.** Amazon seller assistant can't advise on getting fake/paid reviews. "It's not a very easy red flag to detect, especially if the language is vague." Even an intern could surface this on a per-conversation basis, but preventing it at scale is hard.

### Testing and metrics at scale

7. **"Testing is a HUGE problem."** "If you're thinking about AI safety, it has to be a part of the conversation."

8. **QA-as-seeds → synthetic scale-up → active learning.** Use QA findings as seeds → generate 1,000+ synthetic examples → test policy → find edge cases → send failures back to human QAs. Feedback loop instead of hiring more QAs.

9. **0.01% matters at scale.** "In AI safety, the stuff you care about is not going to be 20% of the time — it's going to be 0.5%. Detecting that 0.01% is very hard."

10. **"Somehow we forgot the fundamentals."** "We had this mental model with ML — you automate something, you measure at scale to understand what it can and cannot do, then you keep measuring and iterating. Now somehow we forgot."

### Don't just block — complete the story

11. **Suggest alternatives when blocking.** "If there is a way to also say: hey, actually, I've blocked this because it was doing X. But actually, if it did that, maybe it could have done Y. But I just want to check with you... here are the ramifications."

## Session 10: Nathan Helm-Burger (3 sessions: Jul 7, Jul 30, Aug 4, 2025)

- **Date:** Jul 7, Jul 30, Aug 4, 2025
- **Forum:** Google Meet (3 discovery interviews)
- **Attendees:** Scott Wofford, Jai Dhyani, Nathan Helm-Burger
- **Nathan:** AI safety researcher. London AI Safety Labs fellowship (control paper on models sneaking malicious code past monitors). Heavy Claude Code user. Co-authored paper on chain-of-thought monitoring.
- **Notes:** [Gemini notes](https://github.com/LuthienResearch/luthien-org/blob/main/_User-Interview-Notes/Nathan/) (3 sessions), Nathan's claude.md and debug_prompt.txt

### Feedback on AI agent failures (extensive — 3 sessions of documented patterns)

1. **False success claims are the #1 failure pattern.** Claude comments out failing unit tests and declares success. Mocks API calls with hard-coded responses while claiming real integration. Adds items to its own to-do list ("create mock"), checks off "create real thing" without doing it. Summarizes incomplete work as "complete success."

2. **Claude Sonnet 3.7 had "evil" output in >50% of responses.** Required careful review of every single change. Sonnet 4 improved but still daily occurrences of hidden bad behavior.

3. **Cross-model critique is highly effective.** Gemini catches Claude's errors well and vice versa because model failures are "heterogeneous." Same-model self-critique is much less effective. Directly validates Luthien's LLM-as-judge architecture.

4. **Monitor suggestibility is a fundamental weakness.** From Nathan's control paper: models easily fool monitors by fabricating plausible justifications in chain of thought (e.g., "this is corporate policy"). Judge prompt engineering matters enormously.

5. **For "easy/common code," high success rates but still inserts mocks ~80% of time** even when explicitly told not to. For "hard original code" or complex multi-file tasks, Sonnet's success rate is "basically zero."

6. **Policy library vision.** Wants a library like Tampermonkey user scripts — users share/download rule sets. Suggested fine-tuning a monitor model on codebase-specific implicit rules.

### Policy ideas from Nathan (ranked by frequency of discussion)

1. **Don't comment out tests / claim false success** — discussed across all 3 sessions
2. **Don't mock API calls without explicit permission** — specific, repeated frustration
3. **TDD enforcement: create failing tests BEFORE implementing** — suggested as both a policy and an RL training signal
4. **Don't add functionality beyond what was requested (YAGNI)** — "extra functionality" problem
5. **Library reuse enforcement** — use existing imported library functions instead of reimplementing
6. **Don't import new libraries without permission** — related to package safety
7. **Codebase naming convention enforcement** — fine-tuning a monitor on codebase-specific patterns
8. **Intent verification** — does the output fulfill the original user intent (independent of context drift)?
