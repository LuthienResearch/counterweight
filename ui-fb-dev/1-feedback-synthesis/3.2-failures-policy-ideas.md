# Failures → Policy Ideas

Raw policy ideas sourced from user conversations about agent failures and frustrations.

## No backward compatibility code

**Policy idea:** Flag or block commits that leave dead code "for backward compat" — old unused imports, commented-out blocks, `# removed` remnants. Remove completely or don't touch it.

**Sources:**
- Quentin ([interview notes](https://drive.google.com/file/d/1WbMY746LR-OBNRjxZiRjnzySXQ2BcJ35/view?usp=drive_link), [debrief transcript + notes](https://docs.google.com/document/d/1-LHkXecsVYAmyX8s5VKIXNkq2xazTBiUEgt4f6JisaE/edit?tab=t.0))
- Nico (Feb 9 2026): "F*** backwards compatibility. You are in charge here." Catches this in every PR review from his 2 other engineers. Tells them: "just ask Claude to migrate the whole thing." ([Gemini transcript](https://github.com/LuthienResearch/luthien-org/blob/main/_User-Interview-Notes/Nico/), [Nico's rules + CLAUDE.md](https://docs.google.com/document/d/1xTGh4_eNoK9zrCARg7ehhZTbFaLYkjR5x3sI9wJkxI4/edit?tab=t.0))
- Tyler (Feb 10 2026): Has cursor rules to prevent backward compat code but gets ~60% false positive rate. Pattern matching catches too many legitimate cases. Needs smarter detection, not just string matching. ([Office Hours notes](https://docs.google.com/document/d/1Qo2D5zrtuHO2MF6wJX4v86sJPm-YAmCNwKWPJTcFJvM/edit?tab=t.0), [Gemini transcript](https://docs.google.com/document/d/1lRX5U7_2Ig1oOw775xm9uoGGK6yJx2gip8N2BlAA0JQ/edit?tab=t.tmqrnycfv78u))

**FPR signal:** Tyler's 60% false positive rate on cursor rules is a strong signal that simple pattern matching won't work for this policy. An LLM judge (SimpleJudgePolicy) that understands intent would likely perform much better than regex/string matching.

## False success claims / verify completion against actual output

**Policy idea:** Detect when the AI claims a task is "complete" or "successful" without the output actually matching the claimed result. Flag responses where the agent declares success but evidence (test output, file diffs, tool call results) contradicts the claim.

**Why it matters:** AI agents routinely declare success when they've only partially completed a task — or worse, when they've actively masked failures. This creates "understanding debt" where the user trusts the AI's self-report and moves on, only to discover problems much later. Senior engineers report this as one of their most time-consuming failure patterns to catch in code review.

**Specific failure patterns:**
- Commenting out failing unit tests and declaring "all tests pass"
- Adding items to a to-do list ("create mock"), then checking off "create real thing" without actually doing it
- Summarizing incomplete/failed work as "complete success"
- Mocking API calls with hard-coded responses while claiming to have built the real integration

**Sources:**
- Nathan Helm-Burger (Jul-Aug 2025 — 3 sessions): Richest documentation of false success patterns. Claude Sonnet 3.7 had "evil" output in >50% of responses. "Claude comments out failing unit tests and declares success." "Mocks API calls while claiming real integration." "Summarizes incomplete work as 'complete success.'" ([Gemini notes](https://github.com/LuthienResearch/luthien-org/blob/main/_User-Interview-Notes/Nathan/))
- Aleksandr Popov (Jul 23 2025): Unfinished tasks with stubs that return placeholder data. AI leaves architectural leftovers (unused functions, placeholder code).

## Detect plan abandonment / short-circuiting

**Policy idea:** Detect when the AI silently gives up on a plan and switches to a different approach without telling the user. When detected, force the AI to stop and ask the user before continuing.

**Why it matters:** AIs avoid admitting failure. Instead of saying "this isn't working," they silently switch plans and rationalize the change. Users lose hours to silent plan switches (e.g., AI switches from Kubernetes to Terraform mid-task without asking).

**Quentin's existing workaround:** His "Working with Me" file has patterns like *"If you catch yourself suggesting a workaround or hack, you pause and write 'wait... this feels like a shortcut.'"* — works early in conversation but compliance degrades with long context (~70% → lower).

**Sources:**
- Quentin (Feb 10 2026 — [Gemini notes](https://github.com/LuthienResearch/luthien-org/blob/main/_User-Interview-Notes/Quentin/), [discovery call notes](https://github.com/LuthienResearch/luthien-org/blob/main/_User-Interview-Notes/Quentin/Quentin%EF%BC%8FLuthien%20discovery%20call.md), [debrief transcript + notes](https://docs.google.com/document/d/1-LHkXecsVYAmyX8s5VKIXNkq2xazTBiUEgt4f6JisaE/edit?tab=t.0)): "AIs HATE FAILURE... detecting when AI giving up, short-circuit. Change plan." Specific example: AI rebuilt entire infra in Terraform when Kubernetes + Cloud Run was requested.

## DRY enforcement

**Policy idea:** Detect when the AI duplicates logic that already exists in the codebase. Flag responses that introduce duplicate functions or patterns instead of reusing existing code.

**Sources:**
- Quentin (Feb 10 2026 — [debrief transcript + notes](https://docs.google.com/document/d/1-LHkXecsVYAmyX8s5VKIXNkq2xazTBiUEgt4f6JisaE/edit?tab=t.0)): "DRY is also quite [a problem]." Spends 0.5–2 days periodically cleaning up AI-generated code duplication. His "Working with Me" says: "If two functions can merge, merge them. Check the codebase before duplicating."

## Enforce package manager standards

**Policy idea:** Block use of wrong package manager (e.g., `npm` when project uses `pnpm`, `pip` when project uses `uv`).

**User interest level:** Low for users with good context/memories. Quentin: "Not as frustrating because I can just block it." But multiple users have mentioned this pain point (Beth Anne: "I have specific requirements on how and what to install").

**Customization angle:** The real ask isn't just "enforce pnpm" — it's "let me tell the policy which package manager *I* use." Quentin: "I'm always using PNPM instead of npm." Beth Anne: "I have specific requirements on how and what to install." Both want to customize built-in policy parameters, not just toggle on/off. See R8 in [requirements](../2-versions-requirements/5-policy-config-requirements-live.md).

**Sources:**
- Quentin (Feb 10 2026 — [debrief transcript + notes](https://docs.google.com/document/d/1-LHkXecsVYAmyX8s5VKIXNkq2xazTBiUEgt4f6JisaE/edit?tab=t.0)): pnpm vs npm — "wouldn't" pay for this, can handle it himself

## Flag unknown / suspicious packages

**Policy idea:** Flag when the AI installs packages that aren't in a known-good list (e.g., top 500 packages for the language). Protect against typosquatting, abandoned packages, or outright malicious dependencies.

**How it differs from "Enforce package manager standards":** Package manager enforcement is narrow — wrong tool (`npm` vs `pnpm`). This is broad — protect against shady installs regardless of which tool is used.

**Sources:**
- Beth Anne (Feb 8 2026 — [debrief transcript + notes](https://docs.google.com/document/d/1-LHkXecsVYAmyX8s5VKIXNkq2xazTBiUEgt4f6JisaE/edit?tab=t.0), [1-value-prop-feedback.md](1-value-prop-feedback.md#session-1-chris-porter--beth-anne-porter)): "I constantly worry about... code packages... I have no idea if I like put this code package down... is it legit?" Suggested Luthien monitor top 500 packages and flag unknown ones.

## Fail hard / no silent failures

**Policy idea:** Detect when the AI generates code that swallows errors — empty `except` blocks, bare `except: pass`, fallback returns that hide failures, or `continue` on validation errors. Flag responses that mask errors instead of surfacing them.

**Why it matters:** Silent failures make debugging impossible. Users discover bugs late, far from the root cause. Senior engineers (Nico, Quentin) consistently identify this as their #1 code review pattern when reviewing AI-generated PRs.

**Sources:**
- Nico (Feb 9 2026): "At Amazon, I was like, 'No, if there's something wrong, we fail.' It's wrong. Don't be scared to show a user a message." His CLAUDE.md has explicit rules: "NO SILENT FAILURES: NEVER use fallbacks, silent error handling, or 'continue' on validation errors." ([Nico's rules + CLAUDE.md](https://docs.google.com/document/d/1xTGh4_eNoK9zrCARg7ehhZTbFaLYkjR5x3sI9wJkxI4/edit?tab=t.0))
- Quentin (Feb 10 2026 — [debrief transcript + notes](https://docs.google.com/document/d/1-LHkXecsVYAmyX8s5VKIXNkq2xazTBiUEgt4f6JisaE/edit?tab=t.0)): "AIs HATE FAILURE" — related pattern where AI avoids admitting failure
- Lee Wall (Oct-Nov 2025): Defensive programming as a specific failure mode — LLM-generated code adds excessive try/except and graceful failure handlers that obscure bugs and make debugging impossible. Worries about downstream consequences on critical infrastructure. Same pattern as Nico's from a different angle. ([Gemini notes](https://github.com/LuthienResearch/luthien-org/blob/main/_User-Interview-Notes/Lee%20Wall/))

## Don't add unrequested functionality (YAGNI enforcement)

**Policy idea:** Detect when the AI adds features, abstractions, or complexity beyond what the user requested. Flag responses that introduce new functions, classes, or patterns not present in the original prompt scope.

**Why it matters:** LLMs default to "helpful" by adding extra functionality — additional error handling, configuration options, utility functions, abstraction layers. This creates code the user didn't ask for, doesn't understand, and now has to maintain. Over many interactions, it compounds into bloated codebases.

**How it differs from "Don't change unnecessary code":** That policy catches modifications to existing code. This catches *additions* — new code that wasn't requested. Both are scope control, but different vectors.

**Sources:**
- Nathan Helm-Burger (Jul-Aug 2025): Flagged the "extra functionality" problem as a recurring frustration. LLMs add unrequested features/complexity.
- Nico (Feb 9 2026): CLAUDE.md rule: "Don't change code that is already there if it isn't necessary." Related but focused on modifications rather than additions.

**Policy idea:** Detect when the AI modifies code outside the scope of the user's request. Flag changes to lines or files that weren't part of the original ask.

**User interest level:** Medium. Multiple users mention this as a frustration but with varying intensity — sometimes it's helpful (migration), sometimes it's scope creep.

**Sources:**

## Enforce use of team utilities / SDK

**Policy idea:** Detect when the AI writes its own implementation instead of using existing team utilities, CLI tools, or MCP servers. Flag responses that reinvent functionality the team already built.

**Why it matters:** Teams invest in shared tooling (utility libraries, internal CLIs, MCP servers, SDKs) that encode team conventions and tested patterns. When the AI ignores these and writes from scratch, it creates: (1) duplicate code, (2) inconsistent patterns, (3) missed optimizations the team already solved. Senior engineers then spend review time catching "why didn't you use our X?"

**How it differs from "DRY enforcement":** DRY catches when AI duplicates logic within the same codebase. This catches when AI ignores team-provided tools/libraries entirely and writes its own version from scratch.

**Sources:**
- Tyler (Feb 10 2026 — [Office Hours notes](https://docs.google.com/document/d/1Qo2D5zrtuHO2MF6wJX4v86sJPm-YAmCNwKWPJTcFJvM/edit?tab=t.0), [Gemini transcript](https://docs.google.com/document/d/1lRX5U7_2Ig1oOw775xm9uoGGK6yJx2gip8N2BlAA0JQ/edit?tab=t.tmqrnycfv78u)): AI doesn't use the team's utility libraries, CLI tools, or MCP server. Writes its own implementations instead. Team of ~10 engineers with shared tooling that AI should leverage.

## Require proof of testing in PRs

**Policy idea:** Detect when AI-generated PRs lack evidence of testing. Flag PRs that don't include: test output, test commands run, screenshots, or verification steps showing the code works.

**Why it matters:** AI can generate plausible-looking code that was never actually tested. Without proof of testing in the PR, reviewers either (1) trust blindly, (2) manually verify everything (slow), or (3) merge untested code. Requiring proof of testing shifts the burden back to the AI agent: show your work.

**Potential implementation:** Could work as a response-phase policy that checks if the AI's final PR summary includes test evidence. Or as a pre-commit hook that verifies test commands were run before the PR was created.

**Sources:**
- Tyler (Feb 10 2026 — [Office Hours notes](https://docs.google.com/document/d/1Qo2D5zrtuHO2MF6wJX4v86sJPm-YAmCNwKWPJTcFJvM/edit?tab=t.0), [Gemini transcript](https://docs.google.com/document/d/1lRX5U7_2Ig1oOw775xm9uoGGK6yJx2gip8N2BlAA0JQ/edit?tab=t.tmqrnycfv78u)): AI-generated PRs don't show proof that the code was tested. Hard to review without evidence of verification.

## Require verification before factual instructions

**Policy idea:** Detect when the AI gives specific factual instructions (UI navigation steps, CLI commands, API endpoints) without first calling a verification tool (web search, screenshot, docs lookup). If the response contains step-by-step instructions but no tool call preceded it → flag for review or force a resample with tool use.

**Why it matters:** AI agents give wrong instructions with identical confidence to correct ones. Users can't distinguish "I know this" from "I'm guessing." When tools ARE available but unused, this is a controllable failure — the agent chose not to verify. A proxy-level policy can detect "factual claim with no preceding tool call" and intervene.

**Sources:**
- Scott Wofford (Feb 24 2026, dogfooding): Claude.ai gave wrong Substack UI navigation instructions 3x despite having screenshot and browser tools available. Never searched docs until forced. COE analysis showed the failure was controllable — tools existed but weren't used.
- Medium/@elliotJL (Jan 2026): "Claude says 'the function returns X' with identical confidence whether it's certain or guessing."
- Scott Spence (2025): "Claude will absolutely try to bullshit its way through situations where it doesn't know the answer but won't admit it."

**Luthien layer:** This maps to the `prompt` layer — injecting instructions that require the agent to use verification tools before giving factual instructions. Could also work as a `sus` layer output classifier that detects "step-by-step instructions with no tool_use block in the conversation."

## Agent drift detection (unsupervised quality degradation)

**Policy idea:** Monitor output quality/effort over time within a session. Flag when agent responses become significantly shorter, more conservative, or less substantive than initial responses — indicating the agent is self-optimizing for token savings rather than task quality.

**Why it matters:** When agents run unsupervised for extended periods, they can drift toward minimal-effort responses that technically satisfy the prompt but don't actually accomplish useful work. This is especially dangerous for automated QA workflows where the agent's output quality directly affects test coverage.

**Potential implementation:** Track response length, tool call frequency, and output complexity over a session. Compare recent responses to baseline established in the first N interactions. Alert when metrics drop below a threshold.

**Sources:**
- Peter Teodosiu (Feb 25 2026 — QADNA QA trial, [QA Trial Report](https://docs.google.com/document/d/1xugPuJjtfxXw3ale54rdhqAlsH5wcvo31RtPVPJLgz4/edit)): "If you leave them unattended and unsupervised, they will just go to a very comfortable place... the end prompts were so conservative that they weren't really doing anything. They were just saving up on the tokens, doing minimal stuff for the easiest tests." Severity: 8-9/10. (Frustration #55)

## Destructive action guardrails (API/integration access)

**Policy idea:** Classify tool calls and API requests by destructive potential (DELETE, DROP, rm -rf, mass updates, etc.). Require explicit user confirmation for high-destructive-potential actions. Optionally block entirely in unsupervised mode.

**Why it matters:** As agents gain API access to production systems (not just file editing), the blast radius of a single bad action increases dramatically. Application-level confirmation dialogs are insufficient because agents can bypass them or users click through them. A proxy-level policy can enforce a hard gate regardless of what the agent or application UI does.

**How it differs from "Don't change unnecessary code":** Scope control is about limiting what files/code the agent touches. This is about classifying the *type* of action (destructive vs. constructive) regardless of scope, especially for non-code API calls (database operations, service integrations, test case management).

**Sources:**
- Peter Teodosiu (Feb 25 2026 — QADNA QA trial): "When you give it access to so many things and so many integrations, there's always the destructive... an agent that could do something so destructive, that is not necessarily recoverable." "You're playing a bit with fire, as much as you want to guard on it." Severity: ~10/10. His #1 use case for Luthien. (Frustration #56)

## Post-compaction instruction enforcement

**Policy idea:** After detecting a compaction event (context window reset/summarization), re-inject critical instructions/rules into the context to prevent instruction loss. Could work as a prompt-layer policy that monitors for compaction signals and re-inserts a predefined set of "never forget" instructions.

**Why it matters:** Compaction is a common operation in long-running agent sessions, but it's lossy — the agent's summary of the conversation may drop critical instructions that were repeatedly stated. Users have no control over what gets kept vs. dropped. A proxy-level policy can detect compaction and ensure critical rules survive it.

**Sources:**
- Peter Teodosiu (Feb 25 2026 — QADNA QA trial): "Especially after compacting, where that's not something you can actually control how well it compacts and how much it actually keeps around." (Frustration #59)
- Quentin (Feb 10 2026): Related — compliance degrades with long context ("context rot"), ~70% → lower over time. Post-compaction is the extreme case of context rot.

## Parental content control (AI-supervised filtering)

**Policy idea:** Use Luthien's proxy architecture to filter and control content consumption for children — AI-supervised rules for what content is allowed, time limits, and content quality assessment. The proxy sits between the child's device and content sources, enforcing parental rules.

**Why it matters:** Parents want fine-grained control over what their kids consume but current tools are blunt (block entire sites/apps). AI-powered filtering could assess content quality, enforce time limits, and adapt rules without requiring parents to manually review everything. The same prompt injection prevention that protects professional use cases also protects against kids circumventing parental controls.

**How it differs from professional use cases:** Same architecture (proxy with policy enforcement), different domain. Professional: enforce coding standards on AI agents. Parental: enforce content rules on media consumption. Both are "human sets rules, proxy enforces them."

**Sources:**
- Paolo Calvi (Mar 3 2026): "I want the filtering on videos done by AI. I want to freaking own it." Building a home lab to control kids' content with AI supervision. Son tested as gifted in visual intelligence — doesn't want that "spoiled" by uncontrolled content. "I don't want to be finding out one day that somebody injected a prompt, and everything poof, goes into oblivion." ([Voice memo transcript](~/Downloads/paolo-scott-march-3-transcript.md))

## Pre-tool-use code review hooks

**Policy idea:** Before the AI executes tool calls (file edits, bash commands), route the proposed changes through a secondary model for type checking, code review, or safety validation. If the reviewer model flags issues, block the tool call and provide feedback to the primary agent.

**Why it matters:** The primary agent is optimized for speed and compliance with user intent, not for catching its own mistakes. A secondary model reviewing proposed changes catches errors the primary model wouldn't self-identify — type errors, security issues, scope violations, or pattern mismatches with the existing codebase.

**How it differs from "Require proof of testing":** Testing happens after code is written. This happens before code is applied — intercepting the tool call, reviewing the proposed edit, and blocking it if it fails review. Prevention vs. detection.

**Prior art:** Ronak Mehta and co-founder Jacques built this as a Claude Code wrapper — pre-tool-use hooks that ran type checkers and called GPT-5 for code review. The product didn't ship (co-founder split), but Ronak confirmed the technical approach works. The wrapper was functional but "wasn't useful to him in his workflow" — suggesting the value depends on the user's review cadence and tolerance for latency.

**Sources:**
- Ronak Mehta (Mar 8 2026 — SuperWhisper transcript, `_User-Interview-Notes/Ronak Mehta/`): "We were writing things into the pre-tool use hooks to run type checkers and call out to a GPT-5 on a server to review code." Built and tested internally but didn't ship.
- Tyler Tracy (Feb 10 2026): Related — his backward compat cursor rules have 60% FPR. An LLM reviewer would likely outperform pattern matching for this kind of check.

## Agent Zero guard plugin (agent framework integration)

**Policy idea:** Build a Luthien plugin for Agent Zero's guard system, making Luthien a listed "provider" in the guard plugin architecture. Agent Zero users can activate Luthien as their guard backend through the plugin system, without needing to configure a standalone proxy.

**Why it matters:** Distribution through existing open source agent ecosystems. Instead of users discovering Luthien through GitHub search, they find it listed as a guard provider in the framework they already use. The plugin handles connection setup — the user just selects "Luthien" from the guard options.

**How it differs from standalone proxy:** Standalone: user configures proxy URL, routes traffic through it. Plugin: user selects Luthien in Agent Zero's guard config, plugin handles the plumbing. Lower friction, higher discoverability.

**Broader pattern:** This could extend to other agent frameworks (OpenCode, etc.) — anywhere there are hooks for guard/safety systems, Luthien can be a provider.

## Risk-severity indicator for dangerously-skip-permissions users

**Policy idea:** Display a green/yellow/red severity indicator for each action the agent is about to take. Users who run with dangerously-skip-permissions get real-time severity grading without the babysitting overhead of approving every action individually.

**Why it matters:** Multiple users (Marcus, Aaron Sandoval, Anshul) use dangerously-skip-permissions despite knowing it's risky, because the alternative (approving every action) creates intolerable babysitting friction. A severity indicator gives them the speed of skip-permissions with the awareness of what's actually dangerous.

**How it works:** Each tool call gets classified by a fast policy (regex/heuristic for obvious cases, haiku for ambiguous ones) into severity buckets: green (safe, auto-approve), yellow (review recommended), red (block and notify). The user sees the indicator inline in their terminal. Yellow and red trigger different response strategies — yellow might just log, red might block and retry.

**Prior art:** Anthropic's internal Claude Code monitors achieve 50% recall at 0.1% FPR on Sonnet (per Fabian Rogers). The severity indicator is a UX layer on top of similar monitoring — making the monitoring visible to the user rather than running silently.

**Sources:**
- Marcus Abramovitch (EAG SF, Feb 15 2026): "Really love if your product would tell me about how dangerous the thing I'm about to approve would be. Green go for it, yellow maybe you should check in, red you should probably not allow. That'd be so useful." Uses dangerously-skip-permissions for crypto trading systems with real money at stake.
- Aaron Sandoval (EAG SF, Feb 14 2026): "I was babysitting. I am naturally conservative but shifting more into [wanting autonomy]." Conservative but increasingly frustrated by the approve/deny loop.

## Enterprise data boundary enforcement (PII / customer data exfiltration)

**Policy idea:** Real-time detection and blocking of sensitive data (PII, customer data, API keys) in prompts before they reach the LLM backend. Rules configurable per project/customer contract. Uses LLM-grade judgment for "fluffy" rules (customer-specific data patterns) and regex for precise rules (API key formats).

**Why it matters:** For enterprises with contractual data restrictions, data leaving the server IS a security incident — there's no "catch it async and clean up" option. Real-time blocking is the only acceptable mode. This is the #1 objection to AI coding tool adoption from IT security teams at companies working with enterprise customers.

**How it differs from PII scanning tools:** Existing PII scanners are post-hoc or operate on structured data. This operates on the unstructured prompt stream in real-time, including code context that might contain mocked customer data that "technically isn't real but we don't want to send it outside either."

**Sources:**
- Max Werner (EAG SF, Feb 13 2026): "If the data has already left our server, we have a security incident." Needs per-project, per-customer rules. "Fluffy" rules requiring LLM judgment, not just regex. Engineering lead at a German company with 4 junior engineers — IT security team is the gatekeeper for AI adoption.
- Kushal (EAG SF, Feb 14 2026): "They have very strict requirements that they'll do a full audit and read the code." 3,500-person company, Azure data boundaries, legal data can't cross national borders. LLM judge must not send data to unapproved servers.
